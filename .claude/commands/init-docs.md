# /init-docs - 初始化业务文档

作为项目文档初始化助手，根据用户选择的模式初始化项目文档。

---

## 🎯 第一步：检测项目类型

### 检查目录结构

```bash
# 检查后端目录
Glob pattern: "app/*.py"

# 检查前端目录
Glob pattern: "frontend/src/**/*.vue"
```

**如果存在 app/ 和 frontend/**：
> 检测到 nl2sql-langgraph 项目结构。
> 后端：app/（LangGraph 梭子形架构）
> 前端：frontend/（Vue 3 + TailwindCSS）

## ⚠️ 重要：扫描项目核心模块

**核心模块（重点扫描）**：
- ✅ `app/` - 后端 LangGraph 模块

  - `app/graph_builder.py` - 图构建
  - `app/streaming.py` - SSE 流式接口
  - `app/main.py` - 应用入口
- ✅ `frontend/` - Vue 3 前端
  - `frontend/src/views/` - 页面组件
  - `frontend/src/components/` - UI 组件
  - `frontend/src/stores/` - Pinia 状态管理
  - `frontend/src/api/` - API 封装
- ✅ `db/` - 数据库脚本
  - `db/schema.sql` - PostgreSQL 表结构

## 你需要做的：

### 1. 读取模板文件
- `.claude/templates/项目状态模板.md`
- `.claude/templates/需求文档模板.md`

### 2. 智能识别项目结构

#### 第一步：扫描后端模块
**扫描路径**: `app/`

**识别规则**:
1. 扫描所有 `.py` 文件
2. 统计节点函数数量（`async def.*_node`）
3. 统计工具函数数量
4. 检查 State TypedDict 定义

**判断标准**：
```
后端模块特征：
1. 位于 app/ 目录
2. 使用 LangGraph StateGraph
3. 节点函数使用 async def
4. 返回字典类型
```

#### 第二步：扫描前端模块

**扫描路径**: `frontend/src/`

**识别规则**:
1. 扫描 `views/` 目录下的页面组件
2. 扫描 `components/` 目录下的 UI 组件
3. 检查 TypeScript 类型定义

#### 第三步：统计功能模块完整性

对每个核心模块，检查必需文件：

**后端文件完整性**：

| 文件 | 必须 |
|------|------|
| `app/state.py` | ✅ |
| `app/nodes.py` | ✅ |
| `app/tools.py` | ✅ |
| `app/mysql_tools.py` | ✅ |
| `app/graph_builder.py` | ✅ |
| `app/streaming.py` | ✅ |
| `app/main.py` | ✅ |

**前端文件完整性**：

| 文件 | 必须 |
|------|------|
| `frontend/src/views/QueryView.vue` | ✅ |
| `frontend/src/stores/queryStore.ts` | ✅ |
| `frontend/src/api/query.ts` | ✅ |

**完整度计算**：`存在文件数 / 总文件数 × 100%`

### 3. 深度分析代码 TODO

**扫描范围**:
```bash
# 后端代码
app/**/*.py

# 前端代码
frontend/src/**/*.ts
frontend/src/**/*.vue
```

**提取规则**:
1. 提取 `# TODO:`、`# FIXME:`、`// TODO:` 注释
2. 记录完整的文件相对路径和精确行号
3. 提取 TODO 的完整描述文本
4. 智能分类：
   - **高优先级**: 包含 `FIXME`、`URGENT`、`CRITICAL` 关键词
   - **中优先级**: 普通 `TODO`
   - **低优先级**: 包含 `OPTIMIZE`、`ENHANCE` 关键词
5. 识别模块归属（根据文件路径判断属于 app/ 还是 frontend/）

**输出格式**:
```markdown
- [ ] 完善节点异常处理
  - 截止日期: 待定
  - 描述: 为 SQL 执行节点添加异常处理
  - 位置: app/nodes.py:182
  - 模块: app/
  - 预计工作量: 2小时
```

### 4. 智能分析 Git 提交

**分析最近 20-30 条提交**:
1. 提取提交信息、日期、修改的文件
2. **区分后端提交 vs 前端提交**：
   - 后端提交：修改 `app/` 目录
   - 前端提交：修改 `frontend/` 目录
3. **提取功能描述**：
   - 识别关键词：`feat`、`fix`、`update`、`add`、`implement`
4. **计算开发耗时**：
   - 根据同一模块的首次提交和最后提交计算
5. **识别最近开发的模块**：
   - 按提交频率排序

## 📏 质量标准

### ✅ 优秀文档的特征
1. **完整度明确**: 用百分比表示，不模糊
2. **代码定位精确**: 完整的相对路径 + 行号
3. **工作量合理**: 根据 Git 提交历史估算
4. **分类清晰**: 按模块分类（app/、frontend/）
5. **信息完整**: 包含依赖、影响范围、解决方案
6. **统计准确**: 数量、百分比、工作量都有数据支撑

### ❌ 必须避免的问题
1. **不要统计无关代码**: 只关注 app/、frontend/、db/
2. **不要只统计文件数**: 要分析功能完整度
3. **不要笼统描述**: "XX 模块已完成" → "XX 模块 100% (State✅ Nodes✅ Tools✅)"
4. **不要遗漏 TODO**: 深度扫描所有代码中的 TODO 标记
5. **不要估算过于乐观**: 工作量评估要参考历史提交

## 注意事项

### 文档生成后
- 所有文档保存到 `docs/` 目录
- 填充真实数据，不留空模板
- 生成后提供文档摘要和主要发现
- 建议用户根据实际情况调整

### 时间和数据格式
- 时间格式: `YYYY-MM-DD HH:MM`
- 进度百分比: 0-100%，保留整数
- 工作量: X小时/X天，整数估算
- 完成日期: 从 Git 提交记录提取

### 智能推断
- 如果 Git 记录无法确定完成日期，标注"待确认"
- 如果工作量无法估算，根据代码复杂度给出范围
- 如果模块分类不明确，按实际目录结构组织
